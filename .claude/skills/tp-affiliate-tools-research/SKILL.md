# tp-affiliate-tools-research — аффилиатные инструменты на тревел-блогах (любая страна)

<!-- English copy of this instruction: SKILL.en.md (same folder). -->


Тебе называют **страну** (напр. «по Японии», «Korea», «Таиланд»). Ты:
1. находишь тревел-блоги этой страны (источники — сам, через `WebSearch`);
2. грузишь их через **Playwright** и снимаешь скриншоты **самих элементов-инструментов**;
3. валидируешь (непустые, реальные инструменты, не мусор);
4. **пошагово** дописываешь результат в **HTML-отчёт** `<country>-blogs.html`.

**Вывод ВСЕГДА в HTML.** Никаких финальных «.md как основной результат» — основной артефакт это HTML,
и он должен **расти по мере анализа** (один блог проанализировал → дописал → следующий → дописал).

Всё для работы лежит рядом: `scan-tools.js`, `gen-html.js`, хук `.claude/hooks/validate-html-page.js`.

## 0. Подготовка окружения (один раз)
- Нужен **Playwright**. Если не установлен: `npm run setup` (или `npm install && npx playwright install chromium`).
- Браузер: **по умолчанию HEADLESS на bundled Chromium** (окно НЕ всплывает, стабильно на macOS; важно
  при нескольких параллельных прогонах). Системный Chrome (headed) — только по `HEADED=1`.
- (Опц.) **Playwright MCP** для интерактивного вождения браузера:
  `claude mcp add playwright npx '@playwright/mcp@latest'` — но основной рабочий путь это `scan-tools.js`
  (в нём кастомная логика: element-level скрин, Shadow DOM, валидация — этого MCP «из коробки» не даёт).

## 1. ЧТО ищем — аффилиатные инструменты
Любой элемент, ведущий читателя к бронированию/покупке тревел-услуги (часто через партнёрскую
ссылку/виджет OTA: Booking, Agoda, Klook, KKday, Trip.com, Expedia, GetYourGuide, Viator, Kayak,
Rentalcars, Stay22, Hostelworld, региональные — ezTravel, LionTravel, Jalan, Rakuten и т.п.).

Типы (НЕ исчерпывающе — не отметай незнакомое, если ведёт к бронированию):
- **гиперссылки на OTA** (текст/кнопки), **виджеты** (iframe и **не-iframe**: inline-`<script>`, классы-маркеры),
- **попапы / попандеры / табандеры / интерстишиалы**, **sticky-кнопки/слайдеры**,
- **инлайн-карточки** отелей/туров (фото+цена+рейтинг), **аффилиат-блоки** (мульти-OTA + цены),
- **купоны/промо-страницы** (折扣碼/優惠碼/coupon/promo), **баннеры**,
- **невидимые JS-интеграции** (грузятся скриптом, рендерятся в Shadow DOM): Stay22 `lma`/`letmeallez`,
  **Travelpayouts Emerald** (`emrld.cc`, `<emerald-block>`), Klook/GYG widget JS.

❌ **НЕ инструменты** (сканер их уже отсекает; если что-то проскочило — выкинь при валидации):
video/social-эмбеды (**YouTube/Vimeo/Instagram/Facebook/TikTok**), рекламные слоты (AdSense), соц-кнопки/
подписки, навигация/меню, related-posts. Это не ведёт к бронированию тревел-услуги.

## 2. ЧТО считаем «блогом страны X» (КЛЮЧЕВОЕ определение)
«Тревел-блог Японии / Кореи / Таиланда…» = **локальный блог**: автор **из этой страны и живёт там**,
пишет **на языке страны** (ja/ko/th/zh-TW…) **для местной аудитории** (соотечественников, которые
планируют поездки). Это не «англоязычный гайд о стране для западных туристов».

- ✅ Засчитываем: местный автор, локальный язык, контент для жителей страны (куда съездить, где жить,
  что попробовать) — гайды, маршруты, обзоры отелей/активностей, с монетизацией.
- ⚠️ **Англоязычные expat/digital-nomad/«travel guide для иностранцев» блоги — это НЕ «блог страны»**,
  а блог *про* страну для приезжих. Их можно включать только как **меньшинство** (см. долю в Шаге 3),
  если автор реально местный/живёт там. Не делай отчёт сплошь из них — это нерепрезентативно.
- ❌ Не интересны: чисто справочные сайты без рекомендаций/монетизации, сами OTA/агрегаторы, новостники.
- ❌ **НЕ берём купонники / агрегаторы скидок / «сайты-подборки предложений» (hard-гейт).** Это сайты,
  чьё ЯДРО — лента промокодов/купонов/«deals» по куче брендов (часто не только тревел), а не авторский
  тревел-контент. Симптомы: главная = каталог купонов/скидок по магазинам; нет личного автора/маршрутов/
  обзоров мест; «travel» — лишь одна из категорий рядом с электроникой/одеждой; домен/название вокруг
  *coupon / promo / deals / discount / savings / benefits / zone*. Примеры пойманных нарушителей: **WannaZone,
  benefitshub** — там есть тревел-купоны, но это агрегаторы скидок, не блоги; их попадание **искажает
  результаты** (купоны там — не блогерский инструмент, а сам продукт сайта).
  - 🔑 **Позитивный критерий блога:** на сайте есть **собственный тревел-контент** (гайды, маршруты,
    обзоры мест/отелей/активностей, личные рекомендации «куда съездить / где жить»). Купоны/промо у блога
    допустимы — но **как ОДИН из инструментов монетизации поверх контента**, а не как суть сайта. Нет
    редакционного тревел-контента кроме лент купонов/предложений → это агрегатор, **исключай**.
- ❌ **НЕ берём не-тревел сайты: контент-фермы / MFA («made for affiliate») / сайты на чужой/перепрофилированной
  теме (hard-гейт).** Сайт должен быть **именно про тревел** (тревел — основная тема), а не «общий блог обо
  всём», куда тревел-пост попал случайно. Симптомы: темы вперемешку (смартфоны/电子/финансы/страховки/кредиты
  рядом с тревелом); домен **не имеет отношения к тревелу** (случайная аббревиатура+год, имя конференции,
  истёкший/перекупленный домен); тонкие SEO-посты «під ключевик» (часто свежие пачки) ради аффилиатных
  ссылок/промокодов. Пойманный нарушитель: **`icems2021.com`** (выдавал себя за корейский тревел-блог
  «지식채널», а на деле — контент-ферма на домене конференции ICEMS-2021: посты про Galaxy S23 + промокоды
  Trip.com/Klook). Проверка простая: **открой главную/несколько постов — это автор, пишущий про путешествия,
  или генератор SEO-страниц на случайные темы?** Второе — **исключай**.
- Пограничное — помечай (язык, кто автор, на кого рассчитан), не отбрасывай молча.

🔑 **ТОЛЬКО SELF-HOSTED блоги (жёсткий скоуп).** Нас интересует **собственный сайт автора с CMS**, где он
**сам владеет кодом** и может свободно вставлять виджеты/скрипты/попапы/iframe (WordPress, Ghost, своя
вёрстка, и платформы с кастомным HTML/JS). Именно там живут изучаемые инструменты.
- ❌ **Закрытые платформы-«сады», где нельзя вставить сторонний код — НЕ интересны и НЕ берём:**
  Naver Blog (네이버 블로그), Brunch/Kakao (режет/nofollow внешние ссылки), и любые похожие конструкторы,
  где автор не управляет HTML/скриптами. Там реальных инструментов быть не может — пропускай такие домены.
- Критерий простой: «может ли автор вставить чужой `<script>`/виджет в свою страницу?» Нет → не берём.
  Сомнительно (Tistory, Ameba, note, blogspot, livedoor) — бери, только если видишь, что туда реально
  встроены виджеты/скрипты; иначе предпочитай собственные домены.

Помечай у каждого блога **язык** и тип (местный / англоязычный для иностранцев) — это войдёт в отчёт.
**Классифицируй по ДОМИНИРУЮЩЕМУ языку реального контента страницы**, а не по домену/ожиданию. Бывают
**мультиязычные/пан-региональные** блоги (один сайт отдаёт zh/th/en) — не приписывай их одной стране силой,
помечай «мультиязычный» и учитывай это в выводах (иначе посчитаешь чужой блог за местный — так вышло с itravelblog).

## 3. ИСТОЧНИКИ — как найти блоги страны (Шаг 1)

🛑 **STOP-ГЕЙТ (соблюдать буквально, иначе будет перекос — он повторялся уже трижды):**
- **НЕ запускай ни одного англоязычного `WebSearch` и НЕ строй `targets`-файл, пока не выполнил ≥4 запроса
  на МЕСТНОМ языке и не набрал из них БОЛЬШИНСТВО кандидатов.** WebSearch US/англо-центричен — англ.-первый
  заход гарантированно даёт expat/«guide для иностранцев».
- **НЕ копируй `targets-<другая_страна>.json` как шаблон.** Прошлые файлы могут быть перекошены (напр. Korea
  вышла сплошь англоязычной — это ошибка, не образец). Эталон — тайваньский срез в `examples/` (нормальный
  микс местных). Каждая страна стартует с нуля с поиска на местном языке.
- **Само-проверка ПЕРЕД сканом:** посчитай долю местных vs англоязычных в шортлисте. Для крупных
  локально-язычных рынков местных должно быть **≥60–70%**. Если меньше — ты перекосил: вернись к местным
  запросам и добери, прежде чем запускать сканер.
- Причины, по которым ты обычно соскальзываешь в англ. (помни и сопротивляйся): прецедент в репо;
  US-центричный WebSearch; англ. аффилиат-стек проще детектить. Ни одна не оправдывает нерепрезентативный отчёт.

Алгоритм:
0. **Определи язык(и) страны сам** (ты их знаешь): Япония→ja, Корея→ko, Таиланд→th, Тайвань→zh-TW,
   Исландия→is, Нидерланды→nl и т.д. У маленьких/туристических стран часть местных пишет и по-английски —
   это ок, см. п.3.
1. **Сначала 4–6 запросов на местном языке** — собери из них основную массу.
2. Только потом 1–2 англоязычных запроса — добери немного.
3. **Цель — репрезентативный микс под реальность страны, а не фиксированная цифра.** Для крупных
   локально-язычных рынков (Япония/Корея/Тайвань/Таиланд) это ~70–80% местных. Для маленьких/«англоязычных»
   стран (Исландия, Нидерланды…) доля английского естественно выше — но всё равно ищи **настоящих местных
   авторов** (живут там, пишут для своих), а не международные/expat-travel сайты. Провал = выборка сплошь
   из больших англоязычных «guide for tourists».
4. **Бери ТОЛЬКО self-hosted (свой домен/CMS, автор владеет кодом)** — см. жёсткий скоуп в Шаге 2.
   Закрытые «сады» (Naver Blog, Brunch/Kakao и подобные, где нельзя вставить чужой `<script>`/виджет)
   **пропускай** — там изучаемых инструментов не бывает. Платформу определяй на лету (универсальность —
   Исландия или любая страна): ищешь на местном языке; смотришь, какие **домены повторяются**; отбираешь
   те, где автор реально может встраивать код (собственные домены, WordPress/Ghost; Tistory/blogspot/
   pixnet — если видно встроенные виджеты). Личный домен с WordPress/CMS — приоритет №1.

🎯 **ИЩИ ТРЕВЕЛ-КОНТЕНТ, А НЕ МОНЕТИЗАЦИЮ (главный урок Кореи — не повторять).** Запросы по
монетизационным ключам (`제휴`/`쿠폰`/`할인코드`/`优惠`/`coupon`/`promo code`/`affiliate`) **системно уводят
выборку в купонники и агрегаторы скидок** — у них эти слова в каждом заголовке, они и всплывают первыми.
Ищи **по содержанию**: маршруты, отзывы об отелях, travel-дневники реальных людей, «что посмотреть/где жить».
Инструменты монетизации ты найдёшь УЖЕ ПОТОМ — на самих блогах (Шаг 4), а не через поиск по слову «купон».
Монетизационные ключи допустимы **только** в разведке партнёров (Шаг 3a) — но НЕ как способ искать блоги.

Собери ~12–20 кандидатов; для каждого — **домашний URL** и (желательно) одну **tool-rich seed-статью**
(обзор отелей/маршрут). Шаблоны запросов **на местном языке** (подставь страну):
- Япония: «日本 旅行 ブログ ホテル 予約」, «<город> 観光 ブログ おすすめ», «日本人 旅行ブロガー 人気».
- Корея: «국내여행 블로그 추천 호텔 예약», «<도시> 여행 블로그 후기», «여행 블로거 추천».
- Таиланд: «รีวิว ที่พัก บล็อก เที่ยว <จังหวัด>», «บล็อกเกอร์ ท่องเที่ยว ไทย».
- Тайвань (эталон): «台灣 旅遊 部落格 推薦」, «<城市> 自由行 部落客», «飯店 訂房 部落格».
- англоязычные (добор, меньшинство): «best local <country> travel bloggers», «<country> travel blog (in <language>)».
- зацепки (контентные, не монетизационные): обзоры отелей, маршруты (行程/일정/itinerary), «things to do»,
  подборки, travel-дневники/후기. Платформы выше. ⚠️ Купонные ключи (折扣碼/쿠폰/优惠/coupon) **в поиск блогов
  не добавляй** — они дают купонники; купоны фиксируй уже на найденных тревел-блогах.
Не выдумывай домены — проверяй, что открываются. У каждого кандидата зафиксируй язык и местный/иностранный.

🚦 **Перед добавлением кандидата в `targets-<country>.json` — открой его главную и убедись, что это
действительно ТРЕВЕЛ-блог** (тревел — основная тема, есть авторский контент: гайды/маршруты/обзоры), а **не**:
купонник/агрегатор скидок, контент-ферма/MFA на смешанные темы, перепрофилированный/истёкший домен с
SEO-постами под ключевики (см. hard-гейты в Шаге 2). WebSearch охотно подсовывает такие сайты, потому что у
них есть тревел-пост с промокодом (так в корейский срез попал `icems2021.com` — контент-ферма на домене
конференции). Не доверяй тому, что «нашлось по тревел-запросу» — **верифицируй тему сайта глазами до скана**.

**Локальный монетизац-стек ≠ западный (Booking/Stay22/Klook).** Местные блоги монетизируются через СВОИ
сети/OTA — ищи и детектируй именно их (расширь `OTA_MAP`/`SHORT`, Шаг 3a). Шпаргалка (НЕ исчерпывающе):
- 🇯🇵 **Япония**: сети A8.net, バリューコマース (ValueCommerce), もしも (moshimo), Rakuten アフィリエイト,
  アクセストレード; OTA 楽天トラベル (Rakuten Travel), じゃらん (Jalan), 一休 (Ikyu), JTB, るるぶ; форматы —
  текстовые/баннерные ссылки A8, ракутеновские product-виджеты. (часть уже в `OTA_MAP`/`SHORT`.)
- 🇰🇷 **Корея**: сети 쿠팡파트너스 (Coupang Partners, `link.coupang.com`+`lptag`), 링크프라이스 (LinkPrice),
  텐핑/애드픽; OTA 야놀자 (Yanolja), 여기어때 (GoodChoice), 마이리얼트립, 인터파크, 하나투어, Agoda через
  диплинк `app.ac`. ⚠️ self-hosted: бери **собственные домены**, а Naver Blog/Brunch — пропускай.
- 🇹🇭 **Таиланд**: сети Involve Asia, Accesstrade TH; OTA Agoda/Traveloka; платформа Blockdit.
- 🇹🇼 **Тайвань** (эталон): Klook/KKday/Agoda, Travelpayouts-шортлинки, iChannels 通路王, pse.is/reurl.cc.
- **Любая другая страна** (Исландия и т.д.): локальные сети/OTA свои — выясни их разведкой (Шаг 3a) и
  ищи в местной блогосфере именно их, а не Booking/Stay22.

Сформируй `targets-<country>.json`. **В `name` добавляй язык/тип** (`ja, местный` / `EN, для иностранцев`) —
оно показывается в отчёте, и сразу видно баланс выборки:
```json
[ { "slug": "mimmin", "name": "ミンミンの旅行記 (ja, местный)",
    "home": "https://example.jp/", "seed": "https://example.jp/tokyo-hotels/" },
  { "slug": "nickkembel", "name": "Nick Kembel Travels (EN, для иностранцев)",
    "home": "https://www.nickkembel.com/", "seed": "https://www.nickkembel.com/taiwan-itinerary-1-2-3-weeks/" } ]
```

### 3a. 🚦 ОБЯЗАТЕЛЬНАЯ разведка партнёров под страну (НЕ пропускать — особенно если примеров по стране нет)
Списки `OTA_MAP`/`SHORT` в `scan-tools.js` — это **SEED под Тайвань/Запад/Японию, а не полный реестр**.
Сканер физически не увидит аффилиатку, которой там нет — а у каждой страны свои редиректоры/сети/OTA
(Корея: `app.ac` (диплинк Agoda), `link.coupang.com`+`lptag` (쿠팡파트너스), `linkprice`, 텐핑/애드픽,
야놀자/여기어때/마이리얼트립/하나투어; Исландия: Guide to Iceland и т.п.). **Без этого шага новые страны
СИСТЕМАТИЧЕСКИ недосчитываются** (так и случилось с Кореей). Поэтому ПЕРЕД сканом, ВСЕГДА:
1. `WebSearch` (на местном + en): «<country> travel affiliate programs / networks», «<country> 제휴마케팅 /
   アフィリエイト / 聯盟行銷», «как монетизируются тревел-блоги <country>» — выпиши местные **сети** и **OTA**.
2. Открой 2–3 статьи кандидатов (через WebFetch/Playwright) и посмотри **исходящие внешние домены** и
   **редирект-шортлинки** (повторяющиеся commerce-хосты, ссылки с tracking-параметрами = аффилиатка).
3. **Допиши найденные хосты** в `scan-tools.js`: OTA/тур-платформы → `OTA_MAP` (с именем), редиректоры/
   сети/диплинки → `SHORT`; если у сети особый tracking-параметр (напр. `lptag`, `a8mat`) — добавь его в
   regex `monParams`. Только потом запускай скан.
4. Сомневаешься, аффилиатка ли домен → включи `INCLUDE_UNKNOWN=1` (Шаг 4): он пометит подозрительные
   внешние ссылки как `unknown`, а ты подтвердишь/добавишь их по скриншотам (Шаг 5).
Это и есть механизм работы «в стране без примеров»: язык определяешь сам → партнёров узнаёшь разведкой →
дописываешь в детектор → сканируешь. Не полагайся на seed-список как на полный.

## 4. СКАН + ПОШАГОВАЯ HTML-СБОРКА (Шаг 2) — запусти сканер
```bash
node scan-tools.js targets-<country>.json <country> <locale>
# locale: zh-TW / ja-JP / ko-KR / th-TH / en-US ...
# показать окно браузера (по умолчанию — headless, окно НЕ всплывает):
HEADED=1 node scan-tools.js targets-<country>.json <country> <locale>
# + ловить НЕИЗВЕСТНЫЕ аффилиатки (внешние ссылки с params / повторяющиеся commerce-домены):
INCLUDE_UNKNOWN=1 node scan-tools.js targets-<country>.json <country> <locale>
```
`INCLUDE_UNKNOWN=1` помечает кандидатов типа `unknown` («Внешняя/неизвестная (?) — на проверку»). Это
осознанно «шумный» широкий невод: дальше **ТЫ их валидируешь глазами** (Шаг 5) и либо подтверждаешь
(лучше — дописываешь хост в `OTA_MAP` и пере-сканишь, чтобы тип определился точно), либо выкидываешь.
Включай его, когда нужна полнота / незнакомый рынок (Исландия и т.п.).
**По умолчанию — headless** (окно НЕ всплывает). 🚫 **НЕ переключайся на `HEADED` автоматически** (в т.ч.
при 403/captcha): headed открывает окно браузера и **ломает мульти-агентные/параллельные прогоны** (у
пользователя одновременно несколько стран — окна не должны лезть). 403/блок — просто помечай страницу как
«не удалось проверить» и иди дальше. `HEADED=1` — **только если пользователь явно попросил** показать окно.
Emerald-класс под автоматизацией всё равно не рендерится (нужен CDP, см. ниже).
Сканер сам, на каждый блог: авто-находит до 5 разных страниц (главная + разделы/статьи по
кластерам навигации), грузит, **скроллит до реального низа** (важно: статьи бывают 100k+ px),
ловит инструменты, снимает скриншоты элементов, и **после каждого блога перезаписывает**
`<country>-manifest.json` и `<country>-blogs.html` (вот это «step-by-step»: HTML растёт на глазах).

Покажи пользователю путь к `<country>-blogs.html` сразу — он может открыть и обновлять по ходу.

### Динамические/анти-бот виджеты (Travelpayouts Emerald, Stay22)
- Рендерятся в **Shadow DOM**, грузятся скриптом, **лениво** по скроллу; часть отдаётся только
  доверенной сессии и **под автоматизацией не появляется**. Сканер уже: рекурсивно обходит
  Shadow DOM, медленно скроллит (`slowScroll`), ждёт Emerald, а `page.locator` пробивает open-shadow.
- Если такие блоки нужны как скрин — гони через **CDP к реальному Chrome** (см. README, `CDP_URL=...`).
  Что не поймалось — фиксируется как **JS-интеграция** (мы видим скрипт `emrld.cc`/`travelpayouts`/
  `stay22`), и/или снимается вручную. Не выдумывай — помечай честно.

## 5. ВАЛИДАЦИЯ — два слоя
**(а) В сканере (при захвате)** — чтобы скрин был «инструмента», а не «абы чего»:
- видимость (`display/visibility/opacity`, размер ≥ кнопки, не вся страница);
- **непустой**: файл ≥1.5 КБ И не «большой-но-белый» (отсев по байт-на-пиксель `area>40000 && bytes<area*0.03`);
- содержит текст/`img`/`iframe`/кнопку;
- **отсев не-инструментов**: чужой хром (`nav/header/footer`), меню (≥5 ссылок без OTA/цен/прозы),
  фото-кредиты (`Source:/圖片來源`), соц/комьюнити (`LINE/社群/facebook`), related-posts/pagination,
  внутренние ссылки-воронки (OTA-слово в пути своего домена — матчим OTA по **хосту**), рекламные слоты
  (`adsbygoogle/googlesyndication/doubleclick`).
OTA-ссылка = хост в OTA-списке **или** в шортлинках (`tpk.lu/tp.st/tp-em.cc/emrld.cc/pse.is/reurl.cc/
stay22/i-tm.com.tw`) **или** аффилиат-параметр в query (`aid/cid/marker/trs/tag/...`) при внешнем хосте;
плюс cloaked-редиректы своего домена (`/go/ /recommends/ /aff/`).

**(б) Хук `validate-html-page.js`** (PreToolUse на Write `.html`) — структурный гейт: если ты сам
пишешь HTML-страницу через инструмент Write, передавай тело как JSON `{title, rows:[{"#","Блог",
"Краткое описание","Инструменты","Скриншоты"}]}`; хук проверит, что нет пустых обязательных полей,
и отрендерит таблицу (иначе блокирует Write с причиной). Богатый отчёт со скринами генерит
`gen-html.js` напрямую (минуя хук) — хук это «защита от пустых страниц» для ручных HTML-выводов.

**(в) ТЫ — финальный фильтр шума (главный слой валидации).** Сканер специально настроен на recall
(особенно с `INCLUDE_UNKNOWN=1`), а чистоту наводишь ты по скриншотам. Это не опционально — это часть
процесса:
- **Открой `tools/`-скриншоты** всех `unknown` и любых сомнительных кандидатов (через Read);
- для каждого реши: **(1) реальный booking/аффилиат-инструмент** → подтверди; если это узнаваемый OTA/
  сеть — **допиши хост в `OTA_MAP`/`SHORT` и пере-сканируй** (тогда он получит точный тип, а не `unknown`);
  **(2) шум** (обычная внешняя ссылка, соцсети, реклама, навигация, кредит) → **выкинь** (удали из
  манифеста/исключи фильтром в `scan-tools.js`);
- **в финальный отчёт не должны попадать неподтверждённые `unknown`** — либо переклассифицируй в реальный
  тип, либо убери. Если оставляешь под вопросом — помечай явно «кандидат, не подтверждён».
- Эвристики ошибаются и в обычном режиме — выборочно открывай скрины известных типов тоже; при
  систематических ложняках поправь фильтры в `scan-tools.js` и пере-сканируй.
- 🚫 **Гейт «настоящий тревел-блог» — проверяй на уровне САЙТА, а не отдельного скрина.** Прежде чем оставить
  блог в отчёте, убедись (по главной + типам страниц), что это **авторский тревел-сайт**, а не: (1) агрегатор
  скидок/предложений; (2) контент-ферма/MFA на смешанные/не-тревел темы или перепрофилированный домен
  (см. оба hard-гейта в Шаге 2). Если «блог» — это лента купонов/deals по разным брендам, **или** генератор
  SEO-страниц на случайные темы (смартфоны/финансы/страховки + случайный тревел-пост с промокодом) → **выкинь
  его целиком из манифеста** (не только отдельные скрины). Так в срез просочились **WannaZone / benefitshub**
  (купонники) и **`icems2021.com`** (контент-ферма на домене конференции) — они перекашивают весь результат.
  - 🔔 Сканер сам печатает в лог `⚠ REVIEW <slug>: only N thin link/coupon tool(s)…` для блогов с тонкой
    сигнатурой контент-фермы (только тонкие link/coupon, без виджетов/карточек/cta/интеграций — как у
    icems2021). Это **сигнал к ручной проверке**, не авто-удаление: открой такой сайт и реши по гейтам выше.
    (Купонники-агрегаторы с богатым набором инструментов сканер так НЕ ловит — их отсекаешь ты по теме сайта.)
Итог: широкий невод (B) + твоя глазная валидация = высокая полнота без шума в отчёте.

## 6. ГЕНЕРАЦИЯ — всегда HTML
- Основной отчёт собирает `gen-html.js` (модуль + CLI). Колонки: **# · Блог (имя/URL/типы страниц) ·
  Краткое описание · Инструменты (список по типам + JS-интеграции) · Скриншоты инструментов**.
  Превью **встроены base64** (кросс-платформенный ресайз через Playwright-canvas — без `sips`/`sharp`),
  дедуп на уровне блога. Файл самодостаточный — открывается где угодно.
- Пересобрать вручную: `node gen-html.js <country>-manifest.json <country>-blogs.html "<Title>"`.
- ✅ **Валидный тревел-блог записывай ЧЕСТНО, даже если инструментов ноль.** «0 инструментов» — это
  валидный результат (блог есть, монетизации не нашли), а не повод выкинуть блог или добивать выборку
  купонниками. Не подгоняй отчёт под «побольше инструментов».
- 🗂️ **Купонники/агрегаторы — НЕ в основную таблицу блогов, а в отдельную секцию-наблюдение.** Если по ходу
  попались купонники/агрегаторы скидок (а они попадутся), не считай их блогами и не вычищай молча: вынеси
  **пару показательных примеров со скринами в отдельную секцию «Наблюдение: купонники/агрегаторы»** (внизу
  отчёта) — как контекст рынка, отдельно от авторских тревел-блогов. Так срез остаётся честным и не перекошен.
- (Опц.) суб-отчёт только по купонам с переводом — делай по запросу: отдельный HTML, где каждая
  карточка = скрин + оригинал + перевод (перевод даёшь ты, не хардкодь словарь под одну страну).
